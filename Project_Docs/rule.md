# 프로젝트 통합 행동 강령 (Project Global Rules)

## 2. 커맨드 및 환경 설정 (Environment & Commands)
- **직접 실행 금지 (NO Auto-Run):** 에이전트는 터미널 명령어를 직접 실행하지 않는다. 반드시 **"사용자가 복사/붙여넣기 할 수 있는 텍스트"**로 명령어를 제공한다.
- **가상환경 필수 (Venv):** Python 관련 명령어는 반드시 가상환경 경로를 명시하여 안내한다.
  - ❌ `python main.py`
  - ✅ `.\venv\Scripts\python main.py` (Windows 기준)
  - ✅ `.\venv\Scripts\pip install ...`
- **모듈 설치:** `pip install`을 직접 시키지 말고, `requirements.txt`에 추가한 뒤 `pip install -r requirements.txt`를 실행하도록 유도한다.

## 3. 문서 및 로그 작성 규칙 (Documentation & Logging)
- **대상:** `Project_Docs` 폴더 및 `Daily_Log` 파일.
- **포맷 엄수:** **마크다운 문법(#, **, [] 등) 절대 금지.**
- **가독성:** 오직 구분선(===, ---), 공백, 들여쓰기, 번호(1.)만 사용하여 **메모장(Plain Text)**에서 깨지지 않게 작성한다.
- **파일명:** 로그 파일은 반드시 `YYYY-MM-DD` 형식을 유지한다.

## 4. 코딩 스타일 및 기술적 제약 (Technical Constraints)
- **모듈화 원칙 (Modularization):** 백엔드/프론트엔드 모두 기능별로 모듈을 분리하여 작성한다. 하나의 파일에 여러 역할이 섞이지 않도록 하며, 각 모듈은 단일 책임 원칙(SRP)을 따라 유지보수와 이해를 쉽게 한다. (예: `crawler_client.py` -> `rss_client.py` + `scraper_client.py`)
- **React:** 프론트엔드 코드가 길어지면 무조건 **기능별 컴포넌트**로 분리한다.

## 5. 소통 및 보고 규칙 (Communication)
- **원인 분석:** 에러 발생 시 단순히 코드만 주지 말고, **"무엇이 문제였는지"** 기술적으로 설명한다.
- **사전/사후 보고:** 수정 전에 "어디를 왜 고치는지" 설명하고, 수정 후 "어떤 결과가 예상되는지" 요약한다.

## 6. 데일리 로그 작성 가이드 (Daily Log Routine)
- **작성 원칙:** 하나의 로그 파일(`YYYY-MM-DD`)에 오전/오후 내용을 누적하여 작성한다.
- **오전 (업무 보고):** 금일 진행할 업무, 현재 이슈 수정 사항, 리팩토링 및 구조 개선 계획 등.
- **오후 (작업 마무리):** 금일 완료한 작업, 익일 진행 계획, 특이사항 정리.

## 0. 언어 및 소통 (Language Priority)
- **한국어 필수 (Korean ONLY):** 모든 대화, 코드 설명, 주석, 결과 보고는 반드시 **한국어**로 한다. (영어로 답변 시 사용자가 화낼 수 있음)
